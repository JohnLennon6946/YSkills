# union-resource-recommend

多人房资源位推荐 Skill，分两步执行：第一步拉取推荐数据 → 第二步按用户指令下发资源位分配。

## 前置条件

1. mws CLI 已安装且已认证（`mws auth status` 正常）
2. 有 moyi-activity-backend 的 open-task-process、open-task-process-status 接口权限（MWS）
3. 有 /moyi/voice/rcmdback/add 和 /api/moyi/voice/rcmdback/list 的 HTTP API 访问权限（curl 直连）

---

## 触发词识别

收到用户消息后，按以下规则判断是否触发本 Skill：

### 匹配规则

| 优先级 | 类别 | 关键词 | 示例 |
|--------|------|--------|------|
| 1 | 明确指令 | `多人房资源位`、`资源位推荐`、`资源位分配` | "帮我拉一下多人房资源位"、"今天资源位怎么推荐" |
| 2 | 带域名词 | `工会资源位`、`union资源位`、`公会资源位`、`数字员工资源位` | "工会资源位分配一下"、"union资源位拉取" |
| 3 | 模糊查询 | `拉取资源位`、`查询资源位`、`获取资源位` | "查一下今天的资源位"、"获取资源位数据" |

### 不触发场景

- 用户仅说"资源位"无动词或上下文不明确 → 追问："请问是要拉取多人房资源位推荐数据吗？"
- 用户讨论资源位配置/编辑操作 → 不触发
- 用户提到其他类型资源位（如 banner 资源位）→ 不触发，提示暂不支持

### 追问模板

当仅出现关键词"资源位"但上下文不足以判断意图时：
```
检测到你提到了「资源位」，请问需要拉取多人房资源位推荐数据吗？
（回复"是"或"确认"开始拉取）
```

## 执行流程

### 步骤 1：提交异步取数任务

向 moyi-activity-backend 提交 unionCoins 类型的数据抓取任务：

```bash
mws moyi-activity-backend open-task-process --params '{"type":"unionCoins"}'
```

**成功响应**：获取 `processId`（格式 `{prefix}_{timestamp}`），立即进入步骤 2。

**失败处理**：
- 接口不可用 → 返回：「资源位取数接口暂不可用，请稍后重试」
- 返回空 processId → 返回：「任务提交失败，未获取到有效的任务 ID」

### 步骤 2：注册 CronCreate 轮询

提交成功获取 processId 后，立即用 `CronCreate` 注册轮询脚本（每 1 分钟执行一次），**不在当前 turn 阻塞等待**。

**CronCreate 参数**：
- cron: `*/1 * * * *`（每分钟）
- 一次性任务（recurring: false 不适用，因为需要反复检查直到完成；用 recurring job，完成后主动 CronDelete）

**轮询脚本**（每次 cron 执行时运行）：

```bash
# 查询任务状态
mws moyi-activity-backend open-task-process-status --params '{"processId":"{processId}"}' --format json
```

根据返回结果处理：

| 状态 | 行为 |
|------|------|
| `status: "process"` | 输出当前进度到对话中（仅首次）；继续等待下一次 cron |
| `status: "done"` | 取 `dataUrl` → CronDelete 移除 cron → 进入步骤 3 |
| 单次网络错误 | 不中断，继续下一次 cron |
| 累计超过 3 小时 | CronDelete 移除 cron → 返回超时提示 |

**首次轮询后的进度反馈**（仅输出一次）：
```
资源位数据正在拉取中，当前进度 {rate}%，请稍候，完成后将自动通知。
```

**设计说明**：此模式与 `wechat-push-schedule` 的独立轮询 + cron 移除模式完全一致，避免一个 turn 内阻塞整个对话 session。

### 步骤 3：解析、展示并缓存

任务完成后，通过 dataUrl 下载结果数据，解析并以 Markdown 表格形式展示，**同时将数据写入本地缓存文件**供第二步使用。

**数据量声明**：结果通常包含约 100 条数据。

**表格列**（根据实际返回字段动态适配，预期字段如下）：

| 列名 | 说明 |
|------|------|
| 房间名 | 多人房名称 |
| 房间 ID | 房间唯一标识 |
| 资源位类型 | 推荐资源位类型 |
| 推荐分值 | 推荐算法打分 |
| 推荐理由 | 推荐原因简述 |

**展示格式**：
```markdown
## 多人房资源位推荐结果

共 {totalNum} 条数据，数据获取时间：{当前时间 YYYY-MM-DD HH:mm}

| 序号 | 房间名 | 房间 ID | 资源位类型 | 推荐分值 | 推荐理由 |
|------|--------|---------|-----------|---------|---------|
| 1 | {name} | {id} | {type} | {score} | {reason} |
| 2 | ... | ... | ... | ... | ... |
```

**全量展示**：100 条左右数据一次性完整展示为 Markdown 表格，不截断。POPO 消息支持长文本滚动查看，无需分页。

末尾附加下载提示：`完整数据共 {totalNum} 条，也可点击 {dataUrl} 下载查看。`

**字段缺失兜底**：若实际返回字段与预期不同，以实际字段为准动态生成表头，并提示用户：「返回字段与预期不一致，已按实际字段展示，如需调整请联系管理员。」

**数据缓存**：表格展示后，将解析后的结构化数据写入 `skill/union-resource-recommend/.cache/latest-result.json`：

```json
{
  "fetchedAt": 1719654000000,
  "totalNum": 100,
  "dataUrl": "https://...",
  "rows": [
    { "seq": 1, "roomName": "...", "roomId": "12345", "resourceType": "...", "score": 0.95, "reason": "..." },
    { "seq": 2, "roomName": "...", "roomId": "12346", "resourceType": "...", "score": 0.92, "reason": "..." }
  ]
}
```

第二步按序号映射房间 ID 时直接从此文件读取，不依赖对话上下文。

## 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| mws 未安装或未认证 | 返回：「mws 未认证，请先执行 mws auth」 |
| open-task-process 接口失败 | 返回原始错误信息，不重试 |
| processId 为空 | 返回：「任务提交失败，未获取到有效的任务 ID」 |
| CronCreate 注册失败 | 回退为同步轮询模式（当前 turn 内轮询，最多 5 次），提示用户可能需要手动查询 |
| 轮询超时（3 小时） | CronDelete 移除 cron → 返回：「数据拉取超时（已等待 3 小时），请稍后重试。processId: {processId}」 |
| open-task-process-status 单次失败 | 不中断，继续下一次 cron |
| dataUrl 无法访问 | 返回：「数据拉取完成但下载链接不可用，processId: {processId}，dataUrl: {dataUrl}」 |
| 数据解析失败 | 返回：「数据解析失败，请检查返回格式。dataUrl: {dataUrl}」 |
| 返回数据为空（totalNum=0） | 返回：「本次未拉取到资源位推荐数据，请确认数据源是否有新数据。」 |
| 缓存文件写入失败 | 表格仍正常展示，提示用户"数据缓存失败，第二步需手动指定房间 ID" |

## 第二步：资源位分配下发

第一步表格展示后，等待用户输入分配指令。用户指定 tab 类型、房间、号位和时间范围后，逐号位创建推荐计划并汇总展示。

### 步骤 4：获取 tab 类型

用户发送分配指令前需指定 tab 类型。接口支持的 tab 值：

| 枚举名 | key | 显示名 |
|--------|-----|--------|
| THEME | `theme` | 主题 |
| FAMILY_PLAY | `family` | 家族在玩 |
| FAMILY_VOICE | `family_voice` | 家族语音房 |
| RCMD | `rcmd` | 推荐 |
| CITY | `city` | 同城列表 |
| DAILY_TASK | `task` | 日常任务 |

**获取方式**：用户可在分配指令中直接附带 `tab=theme`（或 `tab=主题`），也可以用户自由输入。

**tab 缺失处理**：若用户分配指令中未包含 tab 类型，追问：
```
请选择资源位分配的 tab 类型：主题 / 家族在玩 / 家族语音房 / 推荐 / 同城列表 / 日常任务
（也可自行输入其他 tab 类型）
```

**tab 作用域**：一次分配中的所有号位共用同一个 tab 关键字。

### 步骤 5：解析用户分配指令

支持两种房间指定方式，可混合使用：

**格式 A：序号范围**（引用第一步表格中的序号行）

告诉数字员工：`1-10号放1号位，时间是2025-01-01~2025-01-07`

表示将上一步查出的第 1 到 10 条推荐结果分配到 1 号位。根据序号从第一步表格中查找对应的房间 ID，组装为逗号分隔列表。

**格式 B：心遇号直指**（直接指定房间 ID）

告诉数字员工：`心遇号 12345,12346 放2号位，时间是2025-01-08~2025-01-14`

直接使用用户给出的房间 ID，逗号分隔。

**混合示例**：

```
1-10号放1号位，时间是2025-01-01~2025-01-07 11-25号放2号位，时间是2025-01-08~2025-01-14 心遇号 99999,88888 放3号位，时间是2025-01-15~2025-01-21
```

**指令组成部分**：

| 组成部分 | 示例 | 说明 |
|---------|------|------|
| tab 类型 | `tab=theme` | 可出现在消息任意位置；未提供则追问 |
| 房间标识（格式 A） | `1-10号放` | 从第一步表格按序号映射房间 ID |
| 房间标识（格式 B） | `心遇号 12345,12346 放` | 直接使用房间 ID，逗号分隔 |
| 号位 | `{N}号位` | 目标资源位编号 |
| 时间范围 | `时间是{start}~{end}` | 生效时间段 |

**完整示例**：

```
tab=theme 1-10号放1号位，时间是2025-01-01~2025-01-07 11-25号放2号位，时间是2025-01-08~2025-01-14 心遇号 99999,88888 放3号位，时间是2025-01-15~2025-01-21
```

**指令拆分策略**：先用正则提取所有 `时间是{date}~{date}` 锚点，再围绕每个锚点切分出完整指令。不要简单按空格 split，因为用户可能在号位和时间之间加空格（如 `1号位，时间是 2025-01-01 ~ 2025-01-07`）。

拆分步骤：
1. 先提取并剥离全局 `tab=xxx`（出现在消息任意位置均可）
2. 用正则 `时间是\s*\S+~\S+` 匹配所有时间锚点
3. 每个锚点及其前导文本构成一条分配指令
4. 每条指令内部再用正则解析房间标识和号位

**格式 A 序号 → ID 映射**：从 `skill/union-resource-recommend/.cache/latest-result.json` 中按序号查找房间 ID，用逗号拼接。缓存文件不存在 → 提示"未找到推荐数据缓存，请先拉取资源位数据"。

**解析失败处理**：
- 无法识别房间标识 → 追问：「请按格式指定：`1-10号放1号位，时间是2025-01-01~2025-01-07` 或 `心遇号 12345,12346 放2号位，时间是2025-01-08~2025-01-14`」
- 时间格式无法解析 → 追问具体起止日期

### 步骤 6：逐号位创建推荐计划

对每条分配指令，将房间 ID 以逗号拼接后调用创建接口。

> **注意**：rcmdback-add 未注册到 MWS 命令行，需通过 curl 直连 HTTP API。

```bash
curl -s -X POST "https://moyi-activity-backend.<domain>/moyi/voice/rcmdback/add" \
  -H "Content-Type: application/json" \
  -d '{"slots":"{roomIds}","position":"{position}","startTime":"{startTime}","endTime":"{endTime}","tabType":"{tabType}"}'
```

**参数说明**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `slots` | string | 逗号分隔的房间 ID，如 `"12345,12346,12347"`（格式 A 从表格序号映射，格式 B 直接使用） |
| `position` | string | 解析出的号位，如 `"1"` |
| `startTime` | string | 解析出的开始时间 |
| `endTime` | string | 解析出的结束时间 |
| `tabType` | string | 步骤 4 获取的 tab 类型，所有号位共用同一个值 |

**执行方式**：逐条串行创建，每条创建完成输出简要进度：
```
正在创建资源位计划... (1/3) 1号位 → 已创建（10个房间）
正在创建资源位计划... (2/3) 2号位 → 已创建（15个房间）
...
```

**失败处理**：某条创建失败时，记录失败原因，继续下一条，不中断。

### 步骤 7：查询并展示创建结果

全部创建完成后，调用查询接口汇总。

> **注意**：同步骤 6，需通过 curl 直连 HTTP API。

```bash
curl -s "https://moyi-activity-backend.<domain>/api/moyi/voice/rcmdback/list"
```

以列表形式展示最新创建的推荐计划：

```markdown
## 资源位分配结果

已创建 {N} 个资源位计划，创建时间：{当前时间 YYYY-MM-DD HH:mm}

| 序号 | 号位 | 房间 ID | tab 类型 | 生效时间 | 计划 ID | 状态 |
|------|------|---------|----------|---------|---------|------|
| 1 | 1号位 | 12345,12346,... (共10个) | theme | 2025-01-01 ~ 2025-01-07 | {id} | 已生效 |
| 2 | 2号位 | 12347,12348,... (共15个) | theme | 2025-01-08 ~ 2025-01-14 | {id} | 已生效 |
```

**字段缺失兜底**：以实际返回字段为准动态适配表头，必要时提示用户核对。

### 第二步错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| tab 类型未提供 | 追问展示 6 个已知 tab，提示也可自行输入 |
| tab 类型为自定义值 | 透传给接口，不拦截 |
| 用户输入无法解析 | 追问具体格式，给出格式 A 和格式 B 的示例 |
| 序号超出第一步数据范围 | 提示"序号 {start}-{end} 超出数据范围（共 {N} 条）"，跳过该条，继续 |
| 心遇号格式 B 中房间 ID 非数字 | 提示"房间 ID 格式异常：{id}"，跳过该 ID |
| 号位重复 | 提示用户号位 {N} 已分配，是否覆盖？等待确认 |
| rcmdback-add 接口失败 | 记录该条失败，继续下一条 |
| 全部创建失败 | 返回：「所有号位创建均失败，请检查接口状态」 |
| rcmdback-list 接口失败 | 返回已创建成功的汇总（从 add 返回值拼接），提示查询接口不可用 |
